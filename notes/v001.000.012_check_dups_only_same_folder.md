## What "same folder" means here

`file_location` in the MUSIQ CSV is the directory part of the relative path: nested dirs get a leading `/` (e.g. `/2024-02-22`), root-level images use `""` (see `musiq.py` and `_get_relative_image_path_from_row` in `duplicates.py`). For any row, that matches **`Path(relative_path).parent`** once you strip the leading slash from `file_location` and join with `file_name`.

So "only compare duplicates within the same `file_location`" is the same rule as: **only compare images whose relative paths share the same parent directory** (e.g. both under `2024-02-22/`, not one in `day1/` and one in `day2/`).

## Where it would go

All pairwise work happens in `find_duplicates_by_score`: the outer loop over keepers and the inner loop over `other_path` (around ```363:396:image_analysis_lib/duplicates.py```). Today every pair (subject to score order, already-assigned dups, optional GPS, and similarity) can match.

You would add an early `continue` when parents differ, e.g. after the index checks and before GPS/similarity:

- `if Path(keeper_path).parent != Path(other_path).parent: continue`

That is equivalent to requiring the same logical folder as `file_location` for paths that come from the same CSV convention.

Optional refinement: **bucket** `paths` by `Path(p).parent` and only run the inner loop within each bucket. Same behavior, less work if you have many folders.

## Extras you might want

- **CLI / config flag** (e.g. `same_folder_only`) so default behavior stays global if anything depends on cross-folder linking.
- **Tests**: There do not appear to be unit tests wired to `find_duplicates_by_score` itself (only helpers like `get_scored_paths_in_order`), so you would mainly add a focused test with mocked encodings if you want coverage.

## Summary

**Difficulty: low** — on the order of a guard (plus optional flag and bucketing). No new data sources are required: paths already encode the folder. The main product decision is whether you still want **any** cross-folder "same scene" links; this change turns that off by definition.

Here the tradeoff is mostly **how many inner-loop iterations** you do, not how many `cosine_similarity` calls (those already drop out if you `continue` on different parents before similarity).

## What bucketing changes

- **No bucketing:** for each of the `n` keepers you scan up to `n` others → **Θ(n²)** iterations of the inner loop (each step does a few cheap checks; many exits early before cosine).
- **Bucketing:** total inner iterations are **Σ nᵢ²** over folders of sizes `n₁, n₂, …`.

If folders are roughly balanced (`nᵢ ≈ n/k`), then  
**Σ nᵢ² ≈ k·(n/k)² = n²/k**  
so you save about a **factor of `k`** on those iterations.

## How many folders matter?

- **At least two folders with images in more than one of them** is where Σ nᵢ² can be **strictly smaller** than n². One folder → one bucket → **no iteration savings** vs the same-folder-only `continue` (cosine count was already like Σ nᵢ²).
- There is no magic threshold like “exactly 7 folders.” What matters is **`n`**, **how uneven the folders are**, and **how much you care about shaving cheap work**.

**Balanced case:** `k` similar-sized folders → rough speedup **~k** on inner-loop work only.

**Very skewed case:** almost everything in one folder → bucketing barely helps (dominant term is still that one big nᵢ²).

## When it is “worth it” in practice

- **For correctness / clarity:** bucketing is optional; a parent check before cosine is enough.
- **For performance:** it starts to matter when **`n` is large enough that the duplicate pass is noticeable** (often hundreds or thousands of images) **and** you have **`k` meaningfully greater than 1`** with a **non-trivial** split (not “10k files in one day folder and 3 stray files elsewhere”).

Rough intuition: if typical day folders have tens to hundreds of photos, **`k` in the tens or hundreds** is exactly where **n² vs n²/k** is the difference between “fine” and “a bit painful,” even though each iteration is cheap.

**Bottom line:** bucketing pays when **Σ nᵢ² ≪ n²** — i.e. **several folders with modest counts each** (or many small folders). It does **not** pay much when **one folder dominates**; **two folders** can already halve inner iterations if sizes are equal, but if one folder has almost everything, do not expect a big win.

It is **not a runtime bug**. It is a **type checker / narrowing** issue.

**What is going on**

1. `gps_radius_meters` is typed as `float | None` on the function (line 319).

2. When it is `None`, you set it from `config.gps_radius_meters`, which is declared as `Optional[float]` in ```26:26:image_analysis_lib/config.py```. So after that block, Pylance still treats `gps_radius_meters` as **`float | None`**.

3. You only enter the GPS distance check when `use_gps` is true (line 391). **At runtime** that means `gps_radius_meters` is not `None` and is `> 0`.

4. Pylance usually **does not** narrow `gps_radius_meters` just because you later test `if use_gps`, since `use_gps` is a separate `bool` and the link to “`gps_radius_meters` is a `float`” is lost. So on line 400 it may still think the right-hand side can be `None`, and then complain about `float > float | None` (wording can look like `>` is not allowed for `float` in some messages).

So: **behavior is fine; the diagnostic is about incomplete narrowing**, not a bad comparison at runtime.

**If you want to silence it** (e.g. in Agent mode): use an explicit guard or local, e.g. `if gps_radius_meters is None: continue` before the `dist` check, or `radius = gps_radius_meters` after asserting or checking it is not `None` inside that branch.
