**Cursor**


Checking the default `min_similarity_threshold` in config; duplicate detection uses cosine similarity, not cosine distance.




In `duplicates.py`, duplicate detection does **not** use a quantity named “cosine distance.” It uses **cosine similarity** from `cosine_similarity()` and marks a pair as duplicate when similarity is **at least** a threshold:

```403:406:image_analysis_lib/duplicates.py
                sim = cosine_similarity(keeper_enc, encodings[other_path])
                if sim >= min_similarity_threshold:
                    duplicate_to_keeper[other_path] = keeper_path
                    keeper_to_duplicates.setdefault(keeper_path, []).append(other_path)
```

That threshold defaults to **`0.65`** via `ImageAnalysisConfig` (unless you pass `min_similarity_threshold` or change config):

```25:25:image_analysis_lib/config.py
    min_similarity_threshold: float = 0.65
```

So in **similarity** terms: duplicates are pairs with **cosine similarity >= 0.65** (default).

If you define **cosine distance** as **`1 - cosine_similarity`** (common for L2-normalized vectors), then the default corresponds to **distance <= 0.35** for a duplicate. The file never computes or logs that distance; it only compares similarity to the threshold.

how good of a value is 0.65?

---

**Cursor**


**0.65 is a reasonable default, not a magic number.** It is a middle setting: strict enough that random different photos are unlikely to hit it, loose enough that “same scene” variants (resized, slightly different crop, close framing) can still match.

**How to read it**

- **Higher (e.g. 0.7-0.85):** Only very similar embeddings count as duplicates. Fewer false positives (different scenes marked as dup), more false negatives (real duplicates missed).
- **Lower (e.g. 0.55-0.6):** More pairs qualify. More aggressive dedup, higher risk of collapsing shots that are only loosely the same scene.

Your README already states that tradeoff: lower threshold = more sensitivity and more false positives; higher = fewer, higher-confidence duplicates.

**Why “good” depends on your data**

The CNN embeddings from `imagededup` define the geometry. Similarity scores are not calibrated like “65% chance duplicate” across all datasets. What matters is how your folders behave: burst shots, bracketing, panoramas, and “same room, two angles” will need different thresholds.

**Practical takeaway**

0.65 is a **sensible starting point** to tune from. If you see obvious different photos marked dup, **raise** it. If obvious near-duplicates are missed, **lower** it slightly and re-run (as the README suggests with e.g. 0.6 vs 0.7). Spot-checking a few folders is the right way to validate it for your pipeline.

if the duplicates.py was run on one folder at a time with different parameters, would it make much difference in the execution time?

---

**Cursor**


Here is a concise read based on how `find_duplicates_by_score` is structured.

## What actually costs time

1. **CNN encodings** - For each image that passes the quality filter, `build_encoding_map_for_paths` builds a `CNN()` and calls `encode_image` once per path. That is usually the dominant cost and scales about **linearly** with how many images you encode.

2. **Similarity checks** - After encodings exist, it is dot products / norms between vectors. That is cheap next to the CNN.

3. **Within-folder only** - Paths are grouped by parent directory; comparisons only happen **inside** each folder. A single full run does not do an all-pairs pass across the whole tree.

So splitting work “one folder at a time” does **not** magically cut total CNN work if you still encode every image once across those runs: total encode time is still roughly proportional to total images.

## Where “one folder at a time” *does* change time

**Separate process / separate CLI run per folder** (typical case):

- Each run constructs `CNN()` again, which usually reloads the model. That fixed cost is paid **once per run**. Doing 20 folders as 20 runs can mean **20 model loads**, which can dominate wall time even if each folder is small.

**One long-lived process** that loops folders:

- Load the model once, then encode folder A, folder B, ... Total model load cost is **once**; total encode time is still ~sum of folder sizes.

## “Different parameters”

If only **`min_similarity_threshold`** (or GPS radius) changes, the **encodings do not change**; only which pairs pass the threshold changes. Re-running the full duplicate step from scratch in a **new process** still recomputes encodings unless you refactor to cache them. So many runs with different thresholds can waste a lot of time on redundant encoding.

## Short answer

- **Same total images, many separate runs:** Often **much slower in wall-clock time** because of repeated CNN/model setup, even though the math work per image is similar.
- **Different parameters:** If you only change threshold, **execution time** is mostly wasted **re-encoding** unless you reuse encodings in one process or add caching.

So yes, it can make a **large** practical difference, mainly from **how many times** you pay model load and encode, not from the similarity loop itself.

## What `duplicates.py` does now

- It builds **one** `encodings` dict for **all** paths that pass the quality filter (every folder at once).
- It then groups paths by parent folder and only **compares** pairs **inside** each folder.

So duplicate **logic** is already per-folder (per bucket). The difference with a “one bucket at a time” design is **when** you build and **how long** you keep those vectors in RAM.

## If you encoded and deduped one folder at a time

- **Peak memory for embeddings** would scale with the **largest** bucket, roughly **O(max folder size)**, instead of **O(N)** for all images under `image_root`.
- You would still need to **encode every image once** across all buckets if you cover the same dataset, so total CNN work is about the same (assuming one process, one model load, sequential buckets).

So yes: **staging encodings per folder and discarding after each bucket would reduce peak RAM** for that embedding store. Other things (path lists, GPS cache for that batch, one copy of the model weights) are usually smaller than “all embeddings for N images” once N is large, but it depends on embedding dimension and N.

## If “one bucket at a time” means one **process** per folder

Peak **embedding** memory drops per process, but each process typically loads the **CNN** again, so you pay **model weights + framework overhead** multiple times. Net RAM on the machine might not go down much, or could go **up** if many runs overlap.

## Given your situation

If RAM is not tight, the main reason to bucket the **pipeline** would be **operational** (progress per folder, crash isolation, different thresholds per folder), not memory. For memory alone, your idea is correct: **holding only the current folder’s encodings lowers peak usage** compared to the current “all encodings in one dict” approach.

Checking how the project pulls in the CNN backend so we can describe what runs at model setup.




There is no fixed number in your repo; **model setup time is almost entirely environment-dependent**. Here is what actually happens and how to think about it.

## What “model setup” means here

`build_encoding_map_for_paths` does `cnn = CNN()` from **imagededup**, which (with your deps) pulls in **TensorFlow** and usually loads a **TensorFlow Hub** CNN (a MobileNet-style encoder). Setup is roughly:

1. **TensorFlow import and runtime init** (often noticeable on its own).
2. **Loading the Hub module / weights** from disk cache, or **downloading** them the first time on that machine.
3. **Building the model in memory** ready for `encode_image`.

That is mostly **fixed overhead per process** (or per `CNN()` instance if you create a new one), not per image.

## Typical ballpark (very rough)

- **Weights already cached, warm disk:** often on the order of **a few seconds to a few tens of seconds** on a laptop CPU before the first `encode_image` runs; can be higher on a Pi or a cold filesystem.
- **First-ever Hub fetch:** can add **much more** (network-bound).
- **GPU:** load/init can still take seconds; encoding then speeds up more than setup.

So for “rerun just some folders,” if each rerun is a **new Python process**, you pay that **again each time**, even if those folders only have a handful of images.

## How this ties to your code

Each call to `find_duplicates_by_score` builds encodings with a **fresh** `CNN()` inside `build_encoding_map_for_paths`. So **one CLI run = one model setup** for that run, then per-image encode.

## Getting a number you can trust

Time **one run** on your machine with **one tiny folder** (or even one image that passes the quality filter). Almost all of that wall time before steady encoding is **TF + Hub + model**. Then time a run with many images; the **extra** time is mostly **encoding**.

So: **rerunning only some folders does not avoid model setup per run** unless you keep a long-lived process and reuse one `CNN()` instance (your library does not expose that today). The cost is “whatever TF + Hub + load is on your box,” not something we can state as a single second value without measuring there.

Yes. That pattern is exactly how you avoid paying **TensorFlow + Hub + model load** on every folder.

**What you get**

- **One** Python process imports TensorFlow and constructs **one** `CNN()` (or whatever loads the weights).
- You read a directory name (stdin, REPL, simple HTTP loop, etc.), run duplicate logic for that bucket, print or write results, then wait for the next name.
- Model setup happens **once** at startup; each folder only pays **encoding + similarity** for the images in that folder.

**What you would need relative to the current library**

Right now `find_duplicates_by_score` always goes through `build_encoding_map_for_paths`, which does `cnn = CNN()` internally. A long-lived driver would want either:

- A **thin API** that accepts an existing `cnn` (or a prebuilt `encodings` dict), or  
- A **small script** that imports lower-level pieces (`build_encoding_map_for_paths` refactored to take optional `cnn`, or duplicate the encode loop in the script).

Without that, each call still creates a new `CNN()` and you lose most of the benefit.

**If you only change the similarity threshold**

Encodings are the same; only `sim >= threshold` changes. In a warm process you could **encode once per folder**, then **sweep several thresholds in memory** with no extra CNN work.

**Practical caveats**

- Your pipeline assumes an `image_root` and MUSIQ CSV layout; each “folder” you pass must still match how paths and CSV rows resolve.
- If you **keep** all folders’ encodings in RAM across iterations, memory grows; if you **drop** after each folder, peak memory stays small.
- Long-running TF processes are normal for this; just handle errors per folder so one bad path does not kill the daemon.

So yes: a warmed process waiting for the next directory (and optionally next threshold) is a sound way to test **per-folder parameters** without repeated model setup.

No. It does **not** need to live in `duplicates.py`, and it usually **should not** if you want a clear split of responsibilities.

**`duplicates.py` today** is library-style code: paths, numpy, CNN, CSV helpers, and `find_duplicates_by_score(...)` with explicit parameters. Nothing in there knows about a CLI or a JSON file on disk.

**Reading `parms.json`** is **configuration and orchestration**: discover files, parse JSON, merge root vs folder overrides, validate keys, decide which folders to loop. That fits best in:

- **`cli.py`** (if it stays small), or  
- A **small dedicated module** (e.g. something like `parms_config.py`) that **`cli.py` imports**, then passes plain values into `duplicates`.

So the flow would be: **CLI (or config helper) reads JSON** → builds per-folder merged params → **calls** `duplicates.find_duplicates_by_score` (or a new helper there that still only takes Python types, not file paths to JSON).

You *could* put `load_parms_json(image_root)` in `duplicates.py` for convenience, but it would tie the library to your config format and file names; most designs keep JSON loading **outside** `duplicates.py` and keep `duplicates.py` as “dumb” functions with explicit arguments.

`cli.py` does not walk anything today because it only ever receives **one** directory and hands it to `duplicates`. If you add a `parms.json` workflow, **something** has to decide the folder list. That can still be `cli.py` (or a tiny helper it calls); it does not have to be `duplicates.py`.

**How it knows where to look for per-folder `parms.json`**

1. **Root `parms.json` names the folders**  
   e.g. `"folders": ["2024-03-01", "trip_a"]`. Then the runner only checks  
   `image_root / name / "parms.json"` for those names. No tree walk required.

2. **Root `parms.json` says “all immediate children”**  
   Then the runner does a **single-level** scan: `for p in image_root.iterdir(): if p.is_dir(): ...`. That is not deep walking the whole image tree; it is just listing day folders (or whatever your top level is).

3. **Deeper overrides**  
   If you ever wanted `parms.json` in nested dirs, you would either list those paths in JSON or do a bounded walk driven by parms (again in CLI/config layer, not in `duplicates.py`).

So: **`duplicates.py` still does not need to discover folders.** The entry layer reads root `parms.json`, builds the list of folder paths (from explicit list or one `iterdir()`), merges each optional `folder/parms.json`, then calls into `duplicates` with concrete paths and parameters.

The important split is: **orchestration** (what folders exist, where JSON lives) vs **dedupe** (encode/compare/write given those inputs). Your observation about `cli.py` is exactly why the new logic belongs in that orchestration layer once you add it, not inside the core duplicate routines.
