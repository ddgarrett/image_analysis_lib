## What creates `_by_status`

The folder is created only inside `copy_images_by_status` in `duplicates.py`: it uses `BY_STATUS_DIR = "_by_status"`, deletes any existing `image_root/_by_status`, recreates it, then copies files into per-status subfolders.

## Ways to ‚Äúremove‚Äù that behavior

1. **CLI only (no code edits)**  
   Run dedupe with **`--no-copy-by-status`**. That skips the whole block above, so **`copy_images_by_status` never runs** and `_by_status` is not created.  
   **Note:** That flag also skips loading MUSIQ rows for that block, so it skips **`write_status_csv`** as well‚Äînot only the folder.

2. **Code change: stop creating the folder but keep status CSV**  
   Keep the `write_status_csv` part, but **remove the `copy_images_by_status` call** (and the ‚ÄúCopied images to ‚Ä¶‚Äù print). You might then **remove or repurpose `--no-copy-by-status`** so it only controls copying, not CSV output, depending on what you want.

3. **Code change: delete the feature**  
   Remove the `copy_images_by_status` call from `cli.py`, then remove **`copy_images_by_status`**, **`BY_STATUS_DIR`**, and **`_STATUS_FOLDER_NAMES`** from `duplicates.py` if unused, plus the CLI flag and README mentions.

**Summary:** The folder is produced only by `copy_images_by_status`. To avoid it without editing code, use `--no-copy-by-status` (knowing it disables the status CSV in that block too). To remove folder generation in code, stop calling `copy_images_by_status` and optionally delete that helper and related constants/CLI/docs.

**User**

just to confirm, are any directories that begin with "_" ignored in both duplicates.py and musiq.py

---

**Cursor**


**`musiq.py` ‚Äî yes.** `find_jpeg_files` skips any path under `root` where **any** path component starts with `.` or `_`, via `_is_ignored_path`:

```40:50:image_analysis_lib/musiq.py
def _is_ignored_path(path: Path, root: Path) -> bool:
    """
    Return True if any component of the path (file or directory)
    starts with '.' or '_', relative to the given root.
    """
    try:
        parts = path.relative_to(root).parts
    except ValueError:
        # If the path is not under root, fall back to absolute parts
        parts = path.parts
    return any(part.startswith(".") or part.startswith("_") for part in parts)
```

**`duplicates.py` ‚Äî not in the same way.** It does **not** walk the tree or call `_is_ignored_path`. It loads **relative paths from the MUSIQ CSV** (`get_scored_paths_in_order` ‚Üí `load_scores_from_musiq_csv`) and uses those. So:

- If duplicates are found using CSVs produced by the normal **`image-analysis score`** / `find_jpeg_files` flow, anything under `_‚Ä¶` was already **excluded at scoring time**, so it **won‚Äôt appear** in the CSV and **won‚Äôt** be in duplicate detection.
- If a CSV **includes** rows for paths under `_‚Ä¶` (manual edit, another tool, etc.), **`duplicates.py` will still use them**; there is no extra `_` filter there.

So: underscore-prefixed directories are **enforced in `musiq.py`** when discovering files; **`duplicates.py` only inherits whatever paths are in the CSV.**
